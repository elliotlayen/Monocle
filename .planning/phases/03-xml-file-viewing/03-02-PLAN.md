    - Store openFile sets parseError to true when DOMParser produces parsererror for XML files

    **XML parser utility (src/features/explorer/utils/xml-parser.ts):**

    Export `ParseResult` interface: `{ document: Document | null; error: string | null }`. Export `parseXml(content: string): ParseResult` function. Use `new DOMParser().parseFromString(content, "text/xml")`. After parsing, check `doc.querySelector("parsererror")` -- if found, return `{ document: null, error: errorNode.textContent ?? "Parse error" }`. If no error, return `{ document: doc, error: null }`. Handle empty string input: check for empty/whitespace-only content before calling DOMParser and return error "Empty content" in that case. Per D-09, this is frontend-only XML parsing. Per Pitfall 1 (RESEARCH.md), DOMParser does NOT throw on malformed XML -- the parsererror check is essential. Per Pitfall 6, `element.tagName` preserves namespace prefixes -- this is correct behavior for display.

    **Store modification (src/features/explorer/store.ts):**

    Import `parseXml` from `../utils/xml-parser`. In the `openFile` action, after successfully reading file content, if `isXml` is true, call `parseXml(content)` and set `parseError: result.error !== null` on the tab. If `parseError` is true, also force `viewMode: "source"` on that tab (per D-10 / CONTEXT.md specifics: malformed XML forces source view). Non-XML files skip parsing; `parseError` stays `false`.

    **Tests (src/features/explorer/utils/xml-parser.test.ts):**

    Follow `date-format.test.ts` vitest pattern. Write RED tests first for all behaviors listed above. Test fixtures: valid XML like `<root><child attr="val">text</child></root>`, malformed XML like `<root><unclosed>`, XML with comments `<!-- comment -->`, XML with CDATA `<![CDATA[data]]>`, XML with PI `<?xml version="1.0"?>`, namespaced XML `<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"><soap:Body/></soap:Envelope>`. Note: vitest runs in jsdom/happy-dom environment which provides DOMParser.

    **XmlTreeView (src/features/explorer/components/xml-tree-view.tsx):**

    Props: `content: string`, `scrollPosition: number`, `onScrollChange: (position: number) => void`. Manage local expand/collapse state with `useState<Set<string>>` (expanded node keys). Parse XML using `parseXml(content)` with `useMemo` to avoid re-parsing on every render (per Pitfall 7 -- parse lazily, only when tree view is rendered). If parse fails, this component should not be rendered (FileContentArea handles that). Generate unique keys for each DOM node by walking the tree depth-first and assigning path-based IDs (e.g., "0", "0.0", "0.1", "0.0.0"). Per D-08, tree starts fully collapsed -- initial `expandedIds` is empty Set, only root element visible. Render root element as the first `XmlTreeNode` at depth 0. Use `ScrollArea` from shadcn for the container: `flex-1 overflow-auto p-2`. On scroll, track position and call `onScrollChange`. On mount, restore scroll position from prop.

    Expand/collapse handlers: `toggleNode(nodeKey: string)` adds or removes from expandedIds Set. When a node is expanded, render its `childNodes` as `XmlTreeNode` components at depth+1. Filter out whitespace-only text nodes (text nodes where `textContent.trim()` is empty) to avoid rendering blank lines.

    **XmlTreeNode (src/features/explorer/components/xml-tree-node.tsx):**

    Props: `node: Node` (DOM Node), `depth: number`, `nodeKey: string`, `isExpanded: boolean`, `onToggle: (key: string) => void`, `expandedIds: Set<string>`. Row container: `flex items-center gap-1 w-full rounded py-1 hover:bg-muted` with `paddingLeft: {depth * 16}px` (same indentation as folder tree per UI-SPEC).

    Render based on `node.nodeType`:

    **Element (nodeType === 1):** Clickable row with expand/collapse. Chevron: `ChevronRight` when collapsed, `ChevronDown` when expanded, hidden (16px spacer) when element has no child nodes. Icon: `FileCode` at h-3.5 w-3.5. Name: `node.tagName` in `text-sm text-foreground`. Attributes: iterate `node.attributes` (NamedNodeMap), render up to 5 inline. Each attribute: name in `text-xs` with XML attribute name color (custom CSS variable or Tailwind arbitrary value matching UI-SPEC `oklch(0.55 0.15 250)` light / `oklch(0.72 0.12 250)` dark), `=` in `text-xs text-muted-foreground`, `"value"` in attribute value color (`oklch(0.55 0.12 150)` light / `oklch(0.72 0.10 150)` dark). Separated by `gap-2`. If more than 5 attributes, show `+N more` in `text-xs text-muted-foreground`. When expanded, render child nodes recursively.

    **Text (nodeType === 3):** Leaf node, no chevron (16px spacer). Icon: `Type` at h-3.5 w-3.5 text-muted-foreground. Content: `text-sm text-foreground truncate max-w-[400px]` showing `node.textContent`. No expand/collapse.

    **Comment (nodeType === 8):** Leaf node. Icon: `MessageSquare` at h-3.5 w-3.5 text-muted-foreground. Content: `text-sm text-muted-foreground italic truncate max-w-[400px]` showing `<!-- {node.textContent} -->`.

    **Processing Instruction (nodeType === 7):** Leaf node. Icon: `Hash` at h-3.5 w-3.5 text-muted-foreground. Content: `text-sm text-muted-foreground truncate max-w-[400px]` showing `<?{node.nodeName} {node.nodeValue}?>`.

    **CDATA Section (nodeType === 4):** Leaf node. Icon: `Braces` at h-3.5 w-3.5 text-muted-foreground. Content: `text-sm truncate max-w-[400px]` in CDATA color (`oklch(0.55 0.08 60)` light / `oklch(0.72 0.06 60)` dark) showing `<![CDATA[{node.textContent}]]>`.

    For the XML semantic colors (attribute names, attribute values, CDATA), use Tailwind's arbitrary value syntax (e.g., `text-[oklch(0.55_0.15_250)]`) or define CSS custom properties and use `dark:text-[...]` for dark mode variants.
