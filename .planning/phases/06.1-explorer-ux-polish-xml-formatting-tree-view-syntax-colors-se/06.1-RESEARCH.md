# Phase 06.1: Explorer UX Polish - Research

**Researched:** 2026-05-28
**Domain:** Frontend UX polish -- React/TypeScript component improvements, XML formatting, tree view styling, UI consistency fixes
**Confidence:** HIGH

## Summary

Phase 06.1 is a UX polish pass on the Integration Explorer feature, addressing six distinct improvement areas identified after completing Phases 1-6. These are all frontend-only changes with no new Rust backend commands or IPC modifications needed. The work spans XML content display, tree view syntax coloring, segmented control consistency, sidebar toggle positioning, date-based tree filtering, and a new "issues only" filter indicator.

Codebase investigation reveals several concrete issues: (1) XML files from network shares are often minified/single-line, making both source view and tree view hard to read without pretty-printing; (2) the XML tree view has syntax colors for attributes and CDATA but element tags (`node.tagName`) use plain `text-foreground` with no XML-specific color; (3) two segmented controls exist (Tree/Source in `file-content-header.tsx` and Filename/Content in `search-bar.tsx`) with inconsistent height, padding, and background opacity; (4) the sidebar toggle button is imported from `schema-graph` (cross-feature coupling) and uses generic screen-reader text; (5) `dateSortOrder` state exists and toggles but is never applied to actual sort logic -- and no date range filter exists; (6) no "issues only" filter exists for the folder tree despite validation badges being present.

A significant finding is that `formatDateFolder()` from `date-format.ts` is defined and tested but never connected to tree node rendering -- date folders show raw names only (e.g., "20251223" without "(Dec 23, 2025)"). This was a Phase 2 success criterion that was implemented as a utility but not wired into the UI.

**Primary recommendation:** All six items are pure frontend work. No new packages are needed -- XML formatting can be done with a simple recursive DOM walk in TypeScript (the DOMParser is already used), and all other items are component-level changes to existing files.

### Pattern 1: XML Pretty-Printing (Client-Side DOM Serialization)

**What:** Format minified XML with proper indentation for readability in source view.
**When to use:** When displaying XML content that arrives as a single line or with inconsistent whitespace.
**Approach:** Use the browser's `DOMParser` (already in `xml-parser.ts`) to parse, then walk the DOM tree to produce indented output. This avoids adding any external dependency.


## Common Pitfalls

### Pitfall 1: XML Formatting Breaks Validation Line Numbers

**What goes wrong:** Pretty-printing XML changes line/column positions, making validation problem locations (line, column) point to wrong positions in the formatted view.
**Why it happens:** Validation runs on the raw content and reports line/column based on original formatting. If source view shows formatted content, the line numbers don't match.
**How to avoid:** Keep a "Format" toggle that reformats for display only. When formatted, either (a) disable click-to-jump from problems panel, or (b) show a notice that line numbers refer to original content. Store both raw and formatted content; validation always runs on raw.
**Warning signs:** User clicks a problem in the problems panel and editor jumps to wrong location.

### Pitfall 2: Date Sort Not Applied to Nested Children

**What goes wrong:** The `dateSortOrder` toggle only affects top-level sort but date folders appear at varying depths (Client > Date).
**Why it happens:** The sort function in `folder-tree.tsx` (line 77) sorts alphabetically. For `YYYYMMDD`-named folders, alphabetical sort happens to match chronological order, so "newest first" just needs reverse alphabetical for folders matching the date pattern.
**How to avoid:** In the sort comparator, detect date folders (8-digit names) and reverse sort order based on `dateSortOrder`. Non-date folders maintain alphabetical sort.
**Warning signs:** User toggles sort direction but nothing visually changes.

### Pitfall 3: Issues-Only Filter with Unloaded Tree Nodes

**What goes wrong:** An "issues only" filter shows nothing because validation data only exists for files that have been opened or scanned.
**Why it happens:** The `validationCache` only contains entries for files that were individually opened or bulk-scanned. Unopened/unscanned files have no validation status.
**How to avoid:** Make the "issues only" filter work with whatever validation data exists (from scans and opened files). Show the filter as available only when there IS validation data. When no data exists for a folder subtree, those nodes pass through (not filtered out).
**Warning signs:** User enables "issues only" and the tree goes blank.

### Pitfall 4: Segmented Control Accessibility Regression

**What goes wrong:** Standardizing the segmented controls breaks keyboard accessibility or ARIA attributes.
**Why it happens:** The Tree/Source toggle in `file-content-header.tsx` uses raw `<button>` elements without `aria-pressed`, while `search-bar.tsx` has `aria-pressed` on its buttons.
**How to avoid:** When standardizing, ensure all segmented controls have: `role="group"`, `aria-label` on container, and `aria-pressed` on each segment button.
**Warning signs:** Screen reader testing shows unlabeled toggle groups.

### Pitfall 5: Performance with Date Filter on Large Trees

**What goes wrong:** Date range filtering re-renders the entire tree on every filter change.
**Why it happens:** The tree filter function creates new node objects, triggering re-renders.
**How to avoid:** Memoize the filtered tree with `useMemo` keyed on the filter parameters. The existing `filterTreeNodes` already follows this pattern. Date filtering should extend it rather than adding a separate pass.
**Warning signs:** Typing in a date range input causes visible lag.

### Example 4: Client-Side XML Pretty-Printing

```typescript
// [ASSUMED] -- standard DOM API approach
export function formatXml(xmlString: string, indentStr: string = "  "): string {
  const parser = new DOMParser();
  const doc = parser.parseFromString(xmlString, "text/xml");
  
  if (doc.querySelector("parsererror")) {
    return xmlString; // Return raw if unparseable
  }
  
  const lines: string[] = [];
  
  // Handle XML declaration if present
  for (let i = 0; i < doc.childNodes.length; i++) {
    const child = doc.childNodes[i];
    if (child.nodeType === 7) { // ProcessingInstruction
      lines.push(`<?${child.nodeName} ${child.nodeValue}?>`);
    }
  }
  
  serializeElement(doc.documentElement, 0, indentStr, lines);
  return lines.join("\n");
}

function serializeElement(
  el: Element,
  depth: number,
  indentStr: string,
  lines: string[]
): void {
  const indent = indentStr.repeat(depth);
  const attrs = Array.from(el.attributes)
    .map(a => `${a.name}="${a.value}"`)
    .join(" ");
  const openTag = attrs ? `<${el.tagName} ${attrs}>` : `<${el.tagName}>`;
  
  const children = Array.from(el.childNodes).filter(
    n => !(n.nodeType === 3 && !n.textContent?.trim())
  );
  
  if (children.length === 0) {
    lines.push(`${indent}<${el.tagName}${attrs ? " " + attrs : ""}/>`);
  } else if (children.length === 1 && children[0].nodeType === 3) {
    lines.push(`${indent}${openTag}${children[0].textContent}</${el.tagName}>`);
  } else {
    lines.push(`${indent}${openTag}`);
    for (const child of children) {
      if (child.nodeType === 1) {
        serializeElement(child as Element, depth + 1, indentStr, lines);
      } else if (child.nodeType === 3) {
        const text = child.textContent?.trim();
        if (text) lines.push(`${indentStr.repeat(depth + 1)}${text}`);
      } else if (child.nodeType === 8) {
        lines.push(`${indentStr.repeat(depth + 1)}<!-- ${child.textContent} -->`);
      } else if (child.nodeType === 4) {
        lines.push(`${indentStr.repeat(depth + 1)}<![CDATA[${child.textContent}]]>`);
      }
    }
    lines.push(`${indent}</${el.tagName}>`);
  }
}
```

## Open Questions

1. **XML Formatting Scope**
   - What we know: Raw XML from network shares is often minified/single-line. Both tree view and source view exist.
   - What's unclear: Should formatting be a persistent toggle per tab, or a global preference? Should the formatted version replace the raw content in the editor, or should it be a separate "Formatted" view mode alongside Tree and Source?
   - Recommendation: Add a "Format" button in the file content header that toggles between raw and formatted in source view. Store the preference per-tab. Keep validation line numbers referencing raw content.

2. **Date Filter Interaction Pattern**
   - What we know: The phase title says "date filter" but no requirements doc specifies the UX.
   - What's unclear: Should it be a date range picker (calendar), preset ranges dropdown, or a text input accepting date patterns? Where does it appear -- sidebar header, above tree, or in a filter bar?
   - Recommendation: A compact dropdown with presets (All dates, Last 7 days, Last 30 days, Last 90 days, Custom range) placed in the sidebar header next to the existing sort toggle. Custom range shows two date inputs.

3. **Issues-Only: Filter or Indicator?**
   - What we know: Validation badges (red/amber dots) already exist on files and folders in the tree. The phase title says "issues-only indicator."
   - What's unclear: Is this a filter that hides clean files, or just an enhanced visual indicator? If a filter, what happens when no validation data exists for a subtree?
   - Recommendation: Implement as a toggle filter button in the sidebar header. When enabled, only show files/folders that have validation issues in the cache. Folders with no validation data pass through (not hidden). The button should show a count of known issues.

4. **Segmented Control: Extract to Shared Component?**
   - What we know: Two segmented controls exist with inconsistent styling.
   - What's unclear: Should we create a reusable `SegmentedControl` component in `src/components/ui/`, or just align the Tailwind classes?
   - Recommendation: Align the Tailwind classes inline for now (two instances don't warrant a shared component). If a third instance appears, extract.
